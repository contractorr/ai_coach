# Agentic Context Compression

**Status:** Draft
**Author:** Raj
**Date:** 2026-03-09

## Problem

The agentic orchestrator has no token awareness. It relies on a hard iteration cap (10) and per-tool output truncation (4000 chars) to stay within context limits. For longer conversations or tools returning large payloads, this means either: (a) hitting the context window limit and getting an API error, or (b) wasting tokens on stale early-turn context that crowds out recent, relevant information.

Reference: [hermes-agent `agent/context_compressor.py`](https://github.com/NousResearch/hermes-agent) — boundary-aligned sliding window with LLM-based summarization of evicted turns, orphaned tool_call/tool_result pair sanitization.

## Users

Users asking complex questions in agentic mode that require multiple tool calls — especially multi-step research, goal coaching with enriched context, or broad "what should I focus on" queries.

## Desired Behavior

1. The orchestrator tracks token usage after each LLM response using the API's `usage` field
2. When accumulated input tokens exceed a configurable threshold (default: 80% of model's context window), the oldest non-protected turns are compressed
3. Compression never splits a tool_call from its corresponding tool_result — they are evicted as a pair
4. The first 2 turns (system context) and last 3 turns are always protected from eviction
5. Evicted turns are summarized into a single "conversation so far" message using `cheap_llm`
6. If summarization fails, evicted turns are silently dropped with a `[earlier context omitted]` marker
7. The user sees no difference in behavior — compression is invisible

## Acceptance Criteria

- [ ] `AgenticOrchestrator` tracks `total_input_tokens` from `response.usage.input_tokens` after each LLM call
- [ ] When tokens exceed threshold, `_compress_context()` is called before the next LLM call
- [ ] Tool call / tool result pairs are never split: if a tool_call message is evicted, its corresponding tool result is also evicted (and vice versa)
- [ ] Orphaned tool results (no matching tool_call) or orphaned tool_calls (no matching result) are cleaned up with stub messages
- [ ] The summary message is generated by `self.cheap_llm` (the existing dual-LLM setup in AdvisorEngine)
- [ ] Summary target is configurable, default 2000 chars
- [ ] If cheap_llm is unavailable or summarization fails, the system falls back to dropping turns with a marker
- [ ] Protected turns: first N messages (configurable, default 2) + last M messages (configurable, default 3) are never evicted
- [ ] Existing agentic tests pass; new tests cover the compression trigger, pair alignment, and fallback behavior

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Conversation has only 5 messages total | Never triggers compression (all protected) |
| Tool call has 3 tool results (parallel calls) | All results evicted together with their tool_call |
| Summary LLM call fails | Evicted turns replaced with `[earlier context omitted — N turns summarized]` |
| Model doesn't return usage in response | Skip compression for that turn, log warning |
| First turn after compression | Summary message placed right after protected-first messages |

## Out of Scope

- Compression for classic RAG mode (single-shot, no multi-turn)
- Caching compressed summaries across sessions
- User-visible "conversation was compressed" notification
- Streaming-aware compression (compression happens between turns, not mid-stream)

## Open Questions

- Should the compression threshold be model-aware (e.g., 80% of 200k for Claude, 80% of 128k for GPT-4o)? The LLM provider interface would need to expose `max_context_tokens`.
